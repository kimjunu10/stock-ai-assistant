# DART 데이터 보완 계획

> 기준: `DART_DATA_INVENTORY_BEFORE.md`의 실제 조사 결과

## 원칙

- 기존 정상 행은 삭제하지 않는다.
- 공시는 `rcept_no`, 재무와 구조화 데이터는 기존 복합 유일키로 upsert한다.
- 원문은 잘렸거나 누락됐거나 중요 유형인데 본문이 없는 공시만 선택한다.
- 임원·주요주주 및 대량보유상황 보고서는 낮은 우선순위로 분리하고 이번 실행에서 전량 수집하지 않는다.
- 뉴스·클러스터링 테이블과 코드는 변경하지 않는다.

## 확인된 차이와 조치

| 차이 | 현재 | 보완 조치 | 완료 조건 |
|---|---|---|---|
| 잘린 원문 | 20건 | `document.xml` 재다운로드, 원본 ZIP 보관, 전체 텍스트 갱신 | 20건 모두 `truncated=false` 또는 실패 사유 기록 |
| 핵심 원문 누락 | 삼성전자 6건 | 우선 재다운로드 | 성공/실패 개별 기록 |
| 중요 공시 원문 누락 | 344건 | 중요도 필터로 증분 수집 | 처리 가능한 대상 0건 |
| 원본 보존 | 없음 | 설정된 로컬 디렉터리에 ZIP 저장 | DB에 상대 경로·길이·해시 존재 |
| 원문 메타데이터 | 없음 | 비파괴 컬럼 추가 | `parse_status`, `parse_error` 포함 |
| 최신 재무 | 2024·2025 | 2026년 공개 보고서 조회, CFS→OFS | 2026년 1분기 등 공개분 upsert |
| 36종 호출 증명 | 이력 없음 | 종목·API·요청범위별 결과 감사 저장 | 180개 호출의 success/no_data/fail 판정 |
| 정기 4종 호출 증명 | 이력 없음 | 종목·연도·보고서·API 결과 저장 | 최신 2개 사업연도 호출 판정 |
| 빈 normalized_data | 1행 | raw 필드의 결정론적 타입 변환 보강 | 재호출 응답이 있으면 비어 있지 않음 |
| 회사 기본정보 | 테이블 없음 | `company.json` 5회, stock_code upsert | 5종목 데이터 존재 |
| 기업 일정 | 테이블 없음 | 중요 일정 공시 원문에서 명시값 파싱 | 후보별 성공/제약 보고 |
| 정정 연결 | 컬럼 없음 | 최초 제출일+정규화 제목으로 유일하게 확인되는 체인만 연결 | 추측 없는 연결 및 미연결 집계 |

## 비파괴 스키마 확장

새 migration에서 다음만 추가한다.

- `disclosures`: 원본 경로, 길이, 해시, 파싱 상태, 원본/정정 연결, 최신본 및 정정 상태
- `company_profiles`
- `corporate_events`
- `dart_collection_runs`
- `dart_collection_api_results`

기존 컬럼이나 행은 삭제·변경하지 않는다. `raw_text_truncated`도 과거 상태 추적과 호환성을 위해 유지한다.

## 수집 실행 단계

1. dry-run으로 대상 접수번호와 예상 API 호출 수만 계산
2. 잘린 20건과 삼성전자 누락 6건 우선 처리
3. 나머지 중요 원문을 종목별로 처리
4. 회사 기본정보 5건 upsert
5. 2025·2026 재무 및 정기보고서 조회
6. 최근 1년 주요사항 36종을 실제 호출하고 결과 상태 기록
7. 일정 후보를 결정론적으로 파싱
8. 정정 체인을 유일하게 확인할 수 있는 경우만 연결
9. 같은 명령을 재실행해 테이블별 행 수 증가 여부 확인

## 실패 정책

- 원문 한 건 실패는 다음 공시를 계속 처리한다.
- API 한 종류 실패는 다음 API를 계속 처리한다.
- HTTP 실패는 최초 호출 후 1회 재시도한다.
- `020`은 설정된 상한까지 지수 백오프한다.
- 인증 오류는 즉시 전체 실행을 중단한다.
- 실패는 `dart_collection_failures.jsonl`과 API 결과 감사 테이블에 동시에 남긴다.

## RAG 범위

이번 작업은 임베딩과 청킹을 실행하지 않는다. 원본 ZIP, 전체 추출 텍스트, 출처, 해시, 길이, 파싱 상태, 최신 정정본 메타데이터까지만 준비한다.
