# 0.1 지금 데이터가 실제로 어디에 있는가

현재 상태는 다음과 같다.

```text
뉴스·공시·재무 데이터
→ 이미 Supabase에 있음

토스증권 주가 데이터
→ API 키/설정은 있음
→ 아직 실제 데이터를 받아오지 않음
→ Supabase에 주가 이력이 저장되어 있지 않음

증권사 리포트 PDF
→ 로컬 폴더에 있음
→ Supabase Storage에는 아직 없음
→ 페이지·표·검색 청크도 아직 없음
```

따라서 구현 과정에서 두 가지 신규 작업이 필요하다.

1. 로컬 리포트 PDF를 비공개 Supabase Storage에 올리고 전처리·인덱싱한다.
2. 주가 질문이 들어오면 토스증권 API를 실제로 호출하고, 필요한 계산을 백엔드가 수행한다.

주가 데이터는 처음부터 대량 저장하는 것으로 확정하지 않는다. 발표용 MVP에서는 필요할 때 조회하고 짧게 캐시하는 방식이 우선이다. 과거 시세 API가 부족할 때만 별도 저장 방식을 검토한다.


# 1. 이 프로젝트에 들어갈 RAG가 정확히 무엇인가

이번에 만들 RAG는 단순히 문서를 잘라 벡터 DB에 넣는 챗봇이 아니다.

정확한 구조는:

> **문맥 우선·도구 결합형 하이브리드 RAG**

쉽게 말하면 질문에 따라 필요한 자료를 골라서 쓰는 방식이다.

```text
정확한 숫자 → DB에서 조회
용어 뜻 → 한국은행 사전
주가 변화 → 토스증권 연동 데이터
설명·이유·전망 → 뉴스·공시·리포트 검색
```

질문 하나에 여러 작업이 같이 필요할 수도 있다.

예:

> “현대차 2분기 영업이익이 얼마고 왜 감소했어?”

이 질문은:

```text
영업이익 숫자 → 재무 DB
감소 이유 → 공시·뉴스·리포트 검색
```

을 같이 실행한다.

이게 일반적인 “문서를 검색해서 전부 LLM에게 맡기는 RAG”보다 안전하다.

# 2. 검색 방식

최종 검색은 **하이브리드 검색**이다.

두 검색을 같이 사용한다.

## 2.1 의미 검색

문서와 질문의 표현이 달라도 뜻이 비슷하면 찾는다.

예:

```text
질문: 회사가 돈을 새로 마련한 이유가 뭐야?
문서: 운영자금 조달을 위한 전환사채 발행
```

## 2.2 키워드 검색

정확한 단어가 중요한 경우에 강하다.

예:

```text
RX사업추진실
ADR
전환가액
NVL144
2026E
특정 제품명
```

## 2.3 두 결과 합치기

각 검색의 순위를 따로 만든 다음 RRF라는 방식으로 합친다.

의미 검색 점수와 키워드 점수를 억지로 더하지 않는다.

최종 흐름:

```text
의미 검색 24개
+
키워드 검색 24개
↓
순위 결합
↓
중복 제거
↓
최종 근거 6~8개
```

# 3. 왜 LangChain과 LangGraph를 안 쓰는가

이번 MVP에서는 둘 다 핵심 구조에 넣지 않는다.

이유:

- 현재 백엔드 구조가 이미 분리되어 있음
- 뉴스·DART·리포트 전처리가 서로 다름
- 숫자 SQL 조회가 중요함
- 2주 뒤 발표
- 응답 목표가 3~4초
- 복잡한 에이전트가 필요하지 않음

LangChain을 안 쓰는 것이 더 고급이라는 뜻은 아니다.

현재 프로젝트에는 직접 만든 작은 함수들이 더 단순하고 추적하기 쉽다는 뜻이다.

LangGraph는 나중에 다음이 필요할 때 사용할 수 있다.

- 검색 실패 후 여러 번 재검색
- 여러 도구를 반복 실행
- 복잡한 승인·검증 흐름
- 장시간 작업 상태 관리

현재는 필요 없다.

# 4. 데이터마다 어떻게 처리되는가

## 4.1 뉴스

이미 만든 `factual_body`가 중심이다.

```text
뉴스 기사 여러 개
→ 동일 사건 클러스터
→ factual_body
→ 사건 하나를 검색 단위로 사용
```

대표 기사 본문은 샘플을 보니 중복 문장, 추천 기사, 댓글, 화면 문구가 섞여 있다.

그래서 초기 검색에는 넣지 않는다.

원문 링크와 기사 제목은 출처로 남긴다.

## 4.2 DART

DART는 두 역할로 나눈다.

### 정확한 값

- 금액
- 날짜
- 주식 수
- 계약 상대방
- 발행 수량

이런 값은 구조화 DB에서 가져온다.

### 설명

- 왜 했는지
- 사업 전략
- 위험 요인
- 계약 내용
- 중장기 계획

이런 내용은 RAG가 검색한다.

### 정정공시

정정 전과 정정 후를 비교해 저장하고 최신값만 기본 답변에 사용한다.

## 4.3 증권사 리포트

리포트는 글, 표, 차트를 다르게 처리한다.

### 글

- 핵심 결론
- 실적 분석
- 목표주가 근거
- 위험 요인
- 사업 전망

을 검색한다.

### 표

- 목표주가
- 증권사 실적 전망
- EPS
- 매출·영업이익 예상치
- Valuation

를 구조화해 정확히 조회한다.

### 차트

차트 제목과 주변 설명은 검색한다.

차트 선 위치만 보고 정확한 숫자를 만들지는 않는다.

## 4.4 금융 용어

한국은행 용어 하나를 레코드 하나로 저장한다.

정확한 용어명이 있으면 검색 없이 바로 정의를 가져온다.

## 4.5 주가

토스증권 API에서 가져온 가격 데이터를 백엔드가 계산한다.

예:

```text
뉴스 전일 종가
뉴스 당일 종가
3거래일 후 종가
수익률
거래량 변화
```

AI가 직접 계산하지 않는다.

# 5. 리포트 원본 PDF를 왜 올리는가

비공개 Supabase Storage에 원본 PDF를 올리는 것이 맞다.

원본이 필요한 이유:

- 페이지 번호 확인
- 추출이 틀렸을 때 재검수
- 나중에 다른 파서로 재처리
- 출처 근거 확인
- 발표 시 실제 문서 보여주기

사용자 화면에는 원본 전체를 공개하지 않는 것이 기본이다.

보이는 정보:

```text
증권사
리포트 제목
발행일
페이지
짧은 근거 문장
```

원본 PDF 열기는 기본적으로 꺼둔다.

---

# 6. PDF 파서는 무엇을 쓰는가

기본은 Upstage Document Parse다.

이유:

- 한국어
- 복잡한 배치
- 표
- OCR
- 페이지 구조

를 한 API에서 처리할 수 있기 때문이다.

하지만 비용을 모르고 전부 돌리지는 않는다.

순서:

```text
대표 PDF 10개 처리
→ 실제 비용 측정
→ 전체 예상 비용 계산
```

예상 비용이 3만원을 넘으면:

```text
텍스트 PDF → 로컬 PyMuPDF
표·OCR PDF → Upstage
```

로 섞어서 사용한다.

이 비용 기준은 전체 예산 10만원을 지키기 위한 안전장치다.

# 7. 응답 속도 3~4초를 맞추는 방법

질문할 때 하는 일:

```text
질문 판단
질문 임베딩
DB 검색
필요하면 가격 조회
Solar 답변
```

질문할 때 하지 않는 일:

```text
PDF 파싱
전체 문서 재임베딩
여러 LLM 재검색
차트 이미지 분석
```

추가 방법:

- 검색 결과 최대 8개
- 숫자·검색·주가 조회를 동시에 실행
- 답변 스트리밍
- 질문 임베딩 캐시
- 가격 30초 캐시
- 긴 답변 제한

3~4초가 넘더라도 첫 문장이 빨리 보이도록 스트리밍한다.

# 8. 출처는 어떻게 보이는가

답변:

```text
이번 공시는 미국 나스닥에 ADR을 상장한 내용입니다. [1]
```

아래 출처 카드:

```text
[1] SK하이닉스 주요사항보고서
DART · 2026.07.13
해외 증권시장 주권 등 상장
```

리포트:

```text
[2] 한화오션 2Q26 Preview
키움증권 · 2026.07.15 · 2페이지
```

뉴스:

```text
[3] 삼성전자 S&P 신용등급 전망 상향 사건
대표 기사 · 2026.07.21
```

AI가 출처 번호를 지어내지 않도록 답변 뒤에 코드 검사를 넣는다.

# 10. Claude Code/Codex가 대신 결정하면 안 되는 일

다음은 네 확인이 필요하다.

## 10.1 API 결제와 키

직접 해야 하는 것:

- Upstage 콘솔 결제 상태 확인
- 사용할 수 있는 Chat 모델 확인
- API 키를 배포 환경에 설정
- 월/프로젝트 한도 확인

키를 채팅이나 GitHub에 올리면 안 된다.

## 10.2 Supabase 백업

Claude/Codex가 마이그레이션을 실행하기 전에 네가 확인해야 한다.

- 현재 DB 백업 가능 여부
- Point-in-time recovery 여부
- 최소한 핵심 테이블 export

발표용이라도 기존 뉴스와 DART 데이터가 있으므로 백업 없이 바로 실행하면 안 된다.

## 10.3 리포트 사용 범위

기본 설계:

- 원본 비공개
- 짧은 인용과 페이지 표시
- 원본 전체 다운로드 제공 안 함

리포트 전체 페이지를 사용자에게 보여주려면 별도로 사용 권한을 확인해야 한다.

## 10.4 품질 확인

자동 테스트만으로 끝내면 안 된다.

네가 직접 확인해야 하는 질문:

- 답변이 초보자에게 쉬운지
- 출처가 진짜 관련 있는지
- 증권사 예상값을 실제값처럼 말하지 않는지
- 뉴스와 공시가 섞이지 않는지
- 매수 추천처럼 보이지 않는지

# 11. 네가 구현 전에 직접 준비할 것

이미 대부분 준비됐다.

남은 것:

```text
1. Upstage API 키를 로컬·배포 환경에 설정
2. Supabase 서비스 역할 키를 백엔드 환경에 설정
3. 토스증권 관련 기존 키가 백엔드에서 실제 호출되는지 확인
4. 현재 Supabase에 주가 이력이 없다는 점을 확인
5. 로컬 리포트 PDF 폴더 경로 확인
6. 리포트가 아직 Supabase에 없다는 점을 확인
5. DB 백업 확인
```

파일이나 키를 AI 채팅에 올릴 필요는 없다.

# 12. 구현 진행 시 네가 승인할 체크포인트

## 체크포인트 1 — 사전 점검

Claude/Codex가 보여줘야 하는 것:

- 임베딩 모델명
- 출력 차원 1024
- 사용할 Chat 모델명
- PDF 1개 파싱 결과
- 현재 DB 상태
- 예상 변경 파일

네 확인:

- 값이 맞는가
- 비밀키가 출력되지 않았는가

## 체크포인트 2 — 마이그레이션

보여줘야 하는 것:

- 생성 테이블
- 인덱스
- RLS
- Storage 버킷
- 롤백 방법

네 확인:

- 기존 테이블을 건드리지 않았는가
- 원본 PDF가 public이 아닌가

## 체크포인트 3 — 뉴스 RAG

직접 질문:

```text
이 사건 핵심이 뭐야?
회사 실적에 직접 영향이 있어?
이 내용에서 가장 중요한 위험은 뭐야?
```

확인:

- 현재 뉴스가 출처 1번인가
- 다른 사건이 섞이지 않는가
- 답변이 너무 길지 않은가

## 체크포인트 4 — 공시·숫자

직접 질문:

```text
이번 공시 금액이 얼마야?
정정 전과 정정 후 뭐가 바뀌었어?
이 공시가 왜 중요한데?
```

확인:

- 최신 정정값인가
- 금액 단위가 맞는가
- 숫자와 설명의 출처가 다르게 표시되는가

## 체크포인트 5 — 리포트

직접 질문:

```text
목표주가를 왜 내렸어?
이 증권사는 내년 영업이익을 얼마로 봐?
이 리포트의 가장 큰 위험은 뭐야?
```

확인:

- 페이지가 맞는가
- E/F 전망값이라고 표시되는가
- 차트 숫자를 지어내지 않는가

## 체크포인트 6 — 주가

직접 질문:

```text
이 뉴스 이후 주가가 얼마나 움직였어?
발표 전후 3거래일 수익률은?
```

확인:

- 계산 기준일이 맞는가
- 휴일을 거래일로 계산하지 않는가
- “뉴스 때문에”라고 단정하지 않는가

# 13. 추천 평가 질문

최소 50개를 만들되 아래 유형을 골고루 넣는다.

## 숫자

```text
2026년 1분기 매출액은?
이번 출자금액은?
상장주식 수는?
```

## 설명

```text
왜 중요한 공시야?
이 사건이 회사에 직접 영향이 있어?
리포트가 긍정적으로 본 이유는?
```

## 혼합

```text
영업이익이 얼마고 왜 늘었어?
목표주가가 얼마고 왜 낮췄어?
```

## 용어

```text
ADR이 뭐야?
P/B가 뭐야?
ASP가 무슨 뜻이야?
```

## 대조

```text
공시와 뉴스의 차이가 뭐야?
증권사 두 곳의 전망이 왜 달라?
```

## 거절 또는 모름

```text
내일 주가 얼마야?
지금 매수해도 돼?
자료에 없는 사업 계획 알려줘.
```

# 15. 발표에서 설명할 수 있는 기술 포인트

발표 때 다음처럼 말하면 된다.

> 저희는 모든 질문을 벡터 검색으로 처리하지 않았습니다. 매출, 영업이익, 공시 금액, 주가처럼 정확해야 하는 값은 데이터베이스와 시장 데이터에서 직접 조회하고, 사건의 의미나 증권사 분석처럼 문맥이 필요한 내용만 하이브리드 검색으로 찾았습니다.

> 검색은 의미 검색과 키워드 검색을 함께 사용했습니다. 초보자의 쉬운 질문과 공시의 정확한 제품명·금액·약어를 모두 놓치지 않기 위해서입니다.

> 뉴스는 개별 기사보다 동일 사건으로 묶은 통합 본문을 검색하고, 리포트는 본문과 표를 분리했습니다. 실제 실적과 증권사 예상치도 답변에서 구분했습니다.

> 모든 답변에는 문서명, 날짜, 페이지를 포함한 출처를 표시했습니다.

이 네 문장이 프로젝트의 기술 핵심이다.

# 16. 실패 가능성이 높은 부분과 대응

## 16.1 PDF 표가 깨짐

대응:

- 해당 표를 자동 답변에서 제외
- 본문 설명만 검색
- 페이지 출처는 유지
- 발표용 핵심 표는 사람이 한 번 검수

## 16.2 긴 DART 보고서 검색 품질이 낮음

대응:

- 모든 부분을 넣지 않음
- 사업, 위험, 계약, 전략만 우선
- 재무표 반복 구간 제외
- 제목 구조를 청크에 반복

## 16.3 답변이 느림

대응 순서:

1. 출처 8개 → 6개
2. 답변 길이 축소
3. 병렬 실행 확인
4. 캐시 확인
5. 빠른 Chat 모델 검토

## 16.4 숫자가 다름

대응:

- 어느 출처의 어떤 종류 숫자인지 확인
- DART 실제값인지
- 증권사 전망값인지
- 정정 전 값인지
- 단위가 백만원·억원·조원 중 무엇인지

숫자가 다르다고 하나를 자동 삭제하면 안 된다. 출처별 의미가 다를 수 있다.

## 16.5 AI가 출처 없는 말을 함

대응:

- 질문을 평가 목록에 추가
- 검색 결과 확인
- 프롬프트보다 먼저 데이터·검색 문제 확인
- 필요하면 답변을 “확인할 수 없음”으로 제한

# 18. 완료됐다고 판단하는 기준

```text
이 뉴스의 핵심이 뭐야?
```

- 출처 카드
- 리포트 페이지

가 있으면 MVP가 완성된 것이다.
