# RAG 평가 계획

## 1. 평가 목적

이 문서는 주식 초보 투자자를 위한 RAG 서비스의 성능을 객관적으로 평가하고, 발표에서 근거 있는 수치를 제시하기 위한 기준을 정리한다.

RAG 성능을 하나의 정확도 숫자로 표현하지 않고 다음 영역을 분리해서 평가한다.

1. 검색 성능
2. 답변 정확성
3. 출처 정확성
4. 숫자·기간·단위 정확성
5. 응답 속도와 비용
6. 답변 불가능 질문 대응

---

## 2. 평가 원칙

### 2.1 개발셋과 홀드아웃셋 분리

- **개발셋**: 검색 방식, 가중치, 프롬프트를 개선할 때 사용
- **홀드아웃셋**: 최종 평가에서만 사용
- 홀드아웃 결과를 보고 코드를 수정했다면 해당 데이터는 더 이상 최종 테스트셋으로 사용하지 않는다.

### 2.2 하드코딩 금지

- 특정 질문, 종목, 기업명, 제품명을 코드에 직접 넣지 않는다.
- 실패 사례별 화이트리스트나 예외 조건을 만들지 않는다.
- 정확 일치, 토큰 정규화, 별칭 테이블, 데이터 빈도, 필터링처럼 전체 데이터에 공통 적용 가능한 방법만 사용한다.

### 2.3 데이터 유형별 결과 분리

다음 데이터를 하나의 점수로 합치지 않는다.

- 뉴스
- DART 공시
- 재무 숫자
- 금융 용어
- 증권사 리포트
- 주가 데이터

각 유형별 검색 성능과 답변 성능을 먼저 측정한 뒤 전체 결과를 요약한다.

---

## 3. 평가 데이터 구성

최종 평가 질문은 약 100~150개를 목표로 한다.

| 유형 | 권장 질문 수 |
|---|---:|
| 뉴스 사건·영향 질문 | 30 |
| 기업명·제품명·약어 검색 | 20 |
| DART 공시 질문 | 20 |
| 재무 숫자 질문 | 20 |
| 증권사 리포트 질문 | 20 |
| 주가·기간 질문 | 10 |
| 모호하거나 답변 불가능한 질문 | 10 |
| **합계** | **130** |

각 평가 질문에는 다음 정답 정보를 미리 기록한다.

```yaml
question: 사용자 질문
question_type: news | dart | finance | report | price | unanswerable
stock_code: 대상 종목코드
gold_document_ids: 정답 문서 ID 목록
expected_facts:
  - 반드시 포함해야 하는 핵심 사실
expected_numbers:
  - value: 정답 숫자
    unit: 단위
    period: 기준 기간
acceptable_sources:
  - 허용 가능한 출처
is_answerable: true
```

## 4. 검색 성능 평가

검색기가 질문에 필요한 문서를 상위 결과에서 찾았는지 평가한다.

### 4.1 Recall@K

정답 문서가 상위 K개 결과 안에 한 개 이상 포함된 질문의 비율이다.

- `Recall@5`
- `Recall@8`

발표에서는 현재 서비스가 답변 생성에 사용하는 검색 개수와 동일한 K를 우선 사용한다.

### 4.2 Hit@1

정답 문서가 검색 결과 1위에 나온 질문의 비율이다.

현재 보고 있는 뉴스나 리포트 화면에서 질문하는 경우에는 해당 문서가 1위 또는 상위권에 들어오는지도 별도로 확인한다.

### 4.3 MRR

정답 문서가 얼마나 높은 순위에 나오는지 측정한다.

- 정답이 1위: 1점
- 정답이 2위: 0.5점
- 정답이 4위: 0.25점
- 정답을 찾지 못함: 0점

모든 질문의 평균을 사용한다.

### 4.4 중복 독점률

같은 사건이나 같은 문서의 유사 청크가 검색 결과를 과도하게 차지하는지 확인한다.

권장 기준:

- 동일 사건·문서가 상위 결과의 25%를 초과하지 않도록 제한
- 예외적으로 문서가 매우 길고 서로 다른 근거가 필요한 경우에는 별도 허용

### 4.5 질문 유형별 분리

검색 성능은 다음 유형으로 나눠 보고한다.

- 종목코드
- 기업명
- 고유 제품명·기술명
- 일반 약어
- 자연어 사건 설명
- 날짜·기간 포함 질문
- 짧고 모호한 토큰
- 현재 화면 문맥이 있는 질문

`AI`, `SK`, `ETF`처럼 코퍼스 전체에 자주 등장하는 표현은 고유명칭 검색과 분리한다.

## 5. 답변 품질 평가

검색된 자료를 바탕으로 올바르고 이해하기 쉽게 답했는지 평가한다.

### 5.1 질문당 5점 체크리스트

| 항목 | 점수 |
|---|---:|
| 질문에 직접 답했다 | 1 |
| 핵심 사실이 원문과 일치한다 | 1 |
| 중요한 내용을 누락하지 않았다 | 1 |
| 검색 근거에 없는 내용을 만들지 않았다 | 1 |
| 초보자가 이해할 수 있는 표현을 사용했다 | 1 |

질문별 점수를 합산하고 평균 점수 및 4점 이상 비율을 보고한다.

### 5.2 근거 충실도

답변의 사실 주장 중 검색된 문서에서 확인할 수 있는 비율이다.

권장 방식:

1. 답변을 검증 가능한 사실 문장 단위로 나눈다.
2. 각 문장을 출처가 직접 뒷받침하는지 확인한다.
3. 근거 있는 문장 수를 전체 사실 문장 수로 나눈다.

### 5.3 핵심정보 포함률

미리 작성한 `expected_facts` 중 실제 답변에 포함된 비율이다.

### 5.4 과도한 단정률

다음과 같은 근거 없는 인과 단정을 실패로 기록한다.

- “이 뉴스 때문에 주가가 올랐다.”
- “이 공시로 앞으로 실적이 반드시 좋아진다.”
- “목표주가가 올라 실제 주가도 상승할 것이다.”

가능성, 관찰 사실, 확인되지 않은 해석을 구분해야 한다.

## 6. 출처 평가

### 6.1 Citation Precision

제시한 출처 중 실제로 답변 내용을 뒷받침하는 출처의 비율이다.

### 6.2 Citation Coverage

출처가 필요한 사실 문장 중 실제 출처가 연결된 문장의 비율이다.

### 6.3 출처 메타데이터 정확성

- 뉴스 제목
- 언론사
- 발행일
- 공시명
- 공시일
- 리포트 제목
- 증권사
- 작성일
- PDF 페이지 번호

위 정보가 원문과 일치하는지 확인한다.

### 6.4 인용 무결성

다음 오류는 별도 집계한다.

- 존재하지 않는 출처 번호
- 답변과 관계없는 출처
- 다른 사건의 출처
- 잘못된 페이지 번호
- 비활성·이전 버전 문서를 최신 근거로 사용

---

## 7. 숫자·기간·단위 평가

금융 서비스에서는 숫자 정확도를 일반 답변 점수와 분리한다.

### 7.1 Exact Match

다음 항목이 모두 맞아야 성공으로 처리한다.

- 숫자
- 단위
- 통화
- 기준 기간
- 연결·별도 기준
- 실제값·전망값 구분
- 최신 정정 여부

예:

- 숫자는 맞지만 `백만원`을 `억원`으로 답함 → 실패
- 2025년 값을 2024년 값으로 답함 → 실패
- 증권사 전망치를 확정 실적으로 표현함 → 실패

### 7.2 권장 지표

- 숫자 Exact Match
- 단위 정확도
- 기간 정확도
- 실제값·전망값 구분 정확도
- 정정공시 최신 버전 선택 정확도

---

## 8. 답변 불가능 질문 평가

자료에 답이 없을 때 억지로 답하지 않는지도 평가한다.

### 평가 항목

- 근거 부족을 명확히 안내했는가
- 관련 없는 문서를 가져와 답하지 않았는가
- 확인할 수 없는 숫자를 생성하지 않았는가
- 필요한 추가 조건을 적절히 요청했는가

### 권장 지표

- 적절한 답변 거절률
- 근거 없는 답변률
- 오답 자신감 비율

---

## 9. 성능과 비용 평가

### 9.1 속도

다음 값을 P50과 P95로 기록한다.

- 검색 지연시간
- 첫 토큰 도착 시간
- 전체 답변 완료 시간
- 외부 API 호출 시간

권장 초기 목표:

| 지표 | 목표 |
|---|---:|
| 검색 P95 | 1초 이하 |
| 첫 토큰 P95 | 2초 안팎 |
| 전체 응답 P95 | 6초 이하 |

### 9.2 안정성

- API 오류율
- 타임아웃 비율
- 인덱싱 실패율
- 재시도 성공률
- 신규 데이터 자동 반영 지연시간
- 중복 임베딩 발생 건수

### 9.3 비용

- 질문 1건당 임베딩 비용
- 질문 1건당 LLM 비용
- 뉴스 신규 인덱싱 비용
- PDF 1개 및 1페이지당 파싱 비용
- 월간 예상 비용

### 9.4 Tool·Agentic 확장 평가 (향후, Extension A~D 대상)

> 아래 지표는 **향후 계획**이다. Extension A~D(공통 Tool·제한형 Agentic·MCP·A2A)가
> 도입될 때 사용한다. 기존 4~9절의 Recall@K·MRR·숫자·출처·성능·비용 지표는 그대로 유지하고,
> 아래를 **추가**로 측정한다. 현재는 결정론적 라우터만 있으므로 미측정 대상이다.

- **Tool 선택 정확도**: 질문이 요구하는 Tool을 라우터·Agent가 올바르게 선택한 비율
- **필수 Tool 누락률**: 정답에 필요한 Tool을 호출하지 않은 비율
- **불필요 Tool 호출률**: 정답에 불필요한 Tool을 호출한 비율
- **반복 호출 횟수**: 동일 Tool을 불필요하게 반복 호출한 정도(무한 루프 방지)
- **복합 질문 완료율**: 다단계·비교 질문을 끝까지 답한 비율
- **fallback 발생률·정확도**: Agent 실패·시간 초과 시 기존 라우터로 fallback한 비율과
  fallback 답변 품질
- **지연시간 비교**: 결정론적 라우터 단독 vs Agentic 경로의 응답 시간
- **비용 비교**: 결정론적 라우터 단독 vs Agentic 경로의 LLM·임베딩 비용

측정 원칙: 단순 질문은 Agent를 거치지 않아야 하므로, 단순/복합 질문을 나눠 각각 위 지표를
분리 집계한다. Agentic 경로가 단순 질문의 지연·비용을 악화시키지 않는지 반드시 확인한다.

## 10. 자동 평가와 사람 평가

### 자동 평가에 적합한 항목

- Recall@K
- Hit@1
- MRR
- 숫자 Exact Match
- 단위·기간 정확도
- 인용 번호 무결성
- 응답 시간
- 비용
- 중복률

### 사람 평가가 필요한 항목

- 초보자가 이해하기 쉬운지
- 핵심 설명이 충분한지
- 지나치게 장황하지 않은지
- 투자 판단을 유도하는 단정이 없는지
- 출처가 실제 주장에 적합한지

사람 평가는 최소 2명이 독립적으로 평가하고, 의견이 다른 질문은 다시 검토하는 방식이 좋다.

## 11. Phase별 평가 연결

### Phase 3

- 뉴스 검색 Recall@8
- 뉴스 검색 MRR
- 고유명칭·자연어 질문 비교
- 의미 검색과 하이브리드 검색 비교

현재 확보한 결과:

| 평가 | 결과 |
|---|---:|
| 뉴스 고유명칭 Recall@8 | 0.92 |
| 별도 홀드아웃 Recall@8 | 0.94 |
| 뉴스 자연어 Recall@8 | 0.975 이상 |

주의: 위 수치는 전체 RAG 정확도가 아니라 **뉴스 검색 성능**이다.

### Phase 4

- DART 검색 성능
- 공시 최신 버전 선택
- 재무 숫자 Exact Match
- 단위·기간 정확도
- 금융 용어 정의 정확도

### Phase 5

- 리포트 검색 성능
- 리포트 페이지 인용 정확도
- 표 숫자 추출 정확도
- 실제값·전망값 구분
- 목표주가·투자의견 추출 정확도

### Phase 6

- 주가 데이터 조회 성공률
- 거래일 기준 계산 정확도
- 기간 수익률 계산 정확도
- 인과관계 과도한 단정률

### Phase 8

- 전체 홀드아웃 평가
- 검색·답변·출처·숫자·속도·비용 종합
- 치명적 오류 사례 분석 및 수정

### Extension A~D (향후)

- §9.4의 Tool·Agentic 지표로 평가한다(Tool 선택 정확도, 필수 Tool 누락률,
  불필요 호출률, 반복 호출, 복합 질문 완료율, fallback, 지연·비용 비교).
- 단순/복합 질문을 분리 집계하고, 결정론적 라우터 대비 회귀가 없는지 확인한다.

## 13. 발표 시 표현 기준

### 사용 가능한 표현

- “뉴스 검색의 고유명칭 Recall@8은 92%였고, 별도 홀드아웃에서 94%를 기록했다.”
- “자연어 뉴스 검색 Recall@8은 97.5% 이상으로 유지됐다.”
- “검색, 답변 사실성, 출처, 숫자 정확도, 속도와 비용을 분리해 평가했다.”
- “평가용 질문을 직접 하드코딩하지 않고 별도 홀드아웃으로 일반화 성능을 확인했다.”

### 사용하면 안 되는 표현

- “전체 RAG 정확도는 94%다.”
- “모든 질문에 94% 정확도로 답한다.”
- “검색 Recall을 답변 정확도라고 표현한다.”
- “뉴스 평가 결과를 공시·리포트·주가에도 동일하게 적용한다.”

## 14. 최종 체크리스트

- [ ] 개발셋과 홀드아웃셋이 분리돼 있다.
- [ ] 질문 유형별 표본 수가 기록돼 있다.
- [ ] 정답 문서와 기대 사실이 사전에 작성돼 있다.
- [ ] 검색 성능과 답변 성능을 분리했다.
- [ ] 뉴스·DART·재무·리포트·주가 결과를 분리했다.
- [ ] 숫자는 단위와 기간까지 Exact Match로 검사했다.
- [ ] 답변 불가능 질문이 포함돼 있다.
- [ ] 출처의 실제 근거 여부를 검사했다.
- [ ] P50/P95 응답 시간을 측정했다.
- [ ] 질문당 비용을 기록했다.
- [ ] 하드코딩과 평가셋 과적합 여부를 확인했다.
- [ ] 발표에서 지표의 대상과 범위를 정확하게 표현한다.
