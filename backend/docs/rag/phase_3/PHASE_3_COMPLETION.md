## recall@8 0.50 원인 분석 → 구조적 개선 (하드코딩 없음)

초기 하이브리드의 정확명칭 recall@8이 0.50에 그친 원인을 유형별로 진단했다.

### 원인 (진단 결과)
1. **평가 질문의 절반이 실제 정확명칭이 아님** — "SK", "AI", "ETF" 같은 초빈도 조각.
   코퍼스에서 "ai"는 1083건, "sk"는 858건에 등장(전체 ~2940건). 특정 사건을 지목할
   변별력이 없어 semantic·lexical 모두 실패하는 게 당연. → 평가 방법론 결함.
2. **lexical 순위가 word_similarity 단일 기준** — 자기 사건이 완전일치(부분문자열 포함)여도
   근사유사 사건들과 뒤섞여 후보 하위로 밀림. → 검색기 구조 개선 여지.

### 개선 A — 검색기(전체 데이터 공통 규칙)
`rag_search_hybrid`의 lexical 순위를 **2단계**로 변경:
`(1) 쿼리 문자열 정확 부분일치(ILIKE) 우선 → (2) word_similarity`.
"쿼리를 그대로 포함하는 청크가 더 관련 높다"는 모든 쿼리 공통 규칙. 특정 값 하드코딩 없음.

> **마이그레이션 이력**: 이 개선은 0017 파일을 고쳐 넣지 않고, 별도 마이그레이션
> `0018_rag_hybrid_lexical_exact_first.sql`에서 `CREATE OR REPLACE FUNCTION`으로 적용한다.
> (0017은 최초 word_similarity 단일 버전으로 유지. DB에 0017이 이미 적용됐던 이력을
> 보존하기 위함.) 롤백 시 0018 down 이 0017 로직으로 함수를 복원한다.

### 개선 B — 평가 방법론(데이터 기반 일반 규칙)
정확명칭 질문 토큰을 "변별력 있는 것"으로 일반화:
- **종목명 조각 제외**: `stocks.name`에서 로드해, 회사명에 포함된 토큰(예: "SK")은
  정확명칭이 아니므로 제외 (특정 회사명 하드코딩 아님, 테이블에서 도출).
- **document-frequency 임계**: 코퍼스 DF 비율이 5%를 넘는 흔한 토큰 제외 (DB로 계산).
- 영문 토큰 최소 길이 3자.
→ 이는 "정확명칭 = 종목명 자체가 아닌, 코퍼스에서 드문 고유표현"이라는 일반 정의.

## 기획서와 달라진 점
1. 키워드 검색을 전체 `similarity(%)` → **word_similarity(<%) + ILIKE**로 구현.
   - 이유: 긴 문서·짧은 쿼리에서 전체 similarity가 threshold 미달로 매칭 0건.
2. lexical 순위를 **정확 부분일치(ILIKE) 우선 → word_similarity** 2단계로.
   - 이유: 완전일치 사건이 근사유사와 뒤섞여 밀리는 문제. 전체 공통 규칙.
3. 부모 문맥은 뉴스에 section이 없어 **앞뒤 청크(chunk_order±1)** 로 확장.
4. `app/rag/fusion.py`(RRF 파이썬 참조) 추가 — 실제 결합은 SQL RPC.
5. 평가 정확명칭 질문 토큰을 종목명 제외 + DF 임계로 일반화(방법론).
